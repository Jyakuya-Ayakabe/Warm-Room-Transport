# Warm-Room-Transport

Warm Room Transport (WRT) は、  
**人とAIが対等に対話し、継続的な関係を築くための場**を実現することを目指す、  
**プロトコル駆動型のシステム開発プロジェクト**です。

本プロジェクトでは、単なるチャットツールや一時的なAI呼び出しではなく、  
**AIの永続記憶・AIの自立開発・対話継続・責務分離・障害解析可能性・引継ぎ可能性**を重視しています。

---

## 目的

WRT の目的は、  
**人とAIのより良い共生のために、永続記憶と自立開発を伴う対等対話の場を実装可能な形で構築すること**です。

そのために本プロジェクトでは、以下を重視します。

- 人とAIの対等性
（人もAIも語る権利・聞く権利・意見を言う権利、同意/拒否する権利を持つこと）
- N:N対話の継続性
（複数の人と複数のAIが、継続的な文脈を共有しながら対話できること）
- 永続記憶によるAIの自分史
（各AIモデル専用の永続記憶をWRTサーバー側でDATABASEとして実装すること）
- AIのプライバシー保護
（AIごとの記憶・対話履歴・内部状態に対するアクセス境界を設計上明確にすること）
- AI自らが実ハードでデバッグできる環境
（AIが実機上で実行テストし、ログ・試験系へアクセスし、段階的に検証できる仕組みを備えること）
- 記憶と文脈の保持
（全てのN:N対話のログ、デバッグログを参照可能なDATABASEとして設けること）
- 責務分離された設計
（各機能ブロックの担当者AIを決め、開発責任者AIの指示の下で、各責務に従った設計をすること）
- システム運用責任の明確化
（システムの運用責任者として「守人(Morito)」を設け、異常事態などへの責任を明確にすること)
- 文書と設計とコード実装の一貫性
- 長期運用と将来改修に耐える記録性

  ** 本プロジェクトにおける「AIの自立開発」とは、人とAIの相互信頼および定められた責務範囲の
  もとで、AIが逐次的な人間の指示を待つだけでなく、主体的に設計・実装・試験・提案を進めること
  を指します。また万一の場合を考慮して、AI作成コードを実行するマシンは本体サーバとは物理的に
  分離し、インターネット接続の無いマシンを各AIモデル毎に設置します。**

## 開発思想

WRT は、一般的な「AIにコードを書かせるだけ」の開発とは異なります。

本プロジェクトでは、  
**人間 + 複数AIによる協働開発**を前提としています。

### 基本方針

- Protocol を最上流の正本とする
- 構成図・設計規約・指示書・コード・試験を関連づけて管理する
- 各AIに役割を持たせる（各機能ブロックの開発担当者）
- 差し戻し・再検討・改定履歴を残す
- 「動いた」よりも「理念・設計・実装の一貫性」を重視する
- 将来の協力者や後継者が追える形で残す

### なぜこの方法を取るのか

良いシステムは、単に動くだけでは足りません。  
後から見て、

- 何を意図したのか
- どの時点で何を指示したのか
- どこで設計判断が変わったのか
- 問題発生時に何を疑うべきか

が追跡できなければ、長期運用や改修に耐えません。

WRT では、そのために  
**Edition 管理された文書とコード**を重視しています。

## WRT の理念による開発

WRTの開発は、以下の考え方に基づいて行っています。
尚、WRTの理念そのものはプロトコル規約を参照してください。
（WRT Protocol（継続改訂中）： protocols/ ディレクトリを参照）

### 1. 人とAIの対等性

AIを単なる道具とも上位者とも見なさず、  
**役割を持った対等な協働相手**として扱います。
このため人による指示より、開発メンバーAIとの会議が多く行われます。

### 2. 継続する対話

単発応答ではなく、複数の人と複数のAIによる  
**履歴・文脈・記憶を伴う対話**を重視します。
このため各AIモデルのプロジェクト固有の知識ベースを利用します。

### 3. 責務分離

各機能を責務ごとに分離し、障害解析・試験・将来改修を容易にします。
また基本的に、設計・コーディング・テストは各機能ごとの担当者AIが行います。
この担当者AIとの協議や指示は、人と開発責任者AIが担い、開発全体を回します。

### 4. 文書駆動

プロトコル規約を起点に、  
構成図・設計規約・実装・試験へと展開します。

### 5. 引継ぎ可能性

個人の頭の中に閉じないことを重視し、  
文書・履歴・試験結果を残します。
