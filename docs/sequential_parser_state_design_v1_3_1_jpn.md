## 1. 設計の目的と位置づけ

本文書は、Protocol Parser の中核実装であるシーケンシャルパーサーのステートマシン設計を定義する。

WRTプロトコル規約（Edition 1.8.2）には以下の要件が明記されている。

> 「通常の一括パース方式（何処かにエラーが有るとその伝文を全て捨てる）ではなく、  
> 逐次パース方式（ステートマシン、1Byte毎に処理し逐次状態遷移し伝文を活かす）様にすること。」

本設計書はこの要件を実装レベルで具体化したものである。

---

### 1-2. なぜステートマシンか ― 将来の実装者へ

**「インターネット（TCP/IP）環境ならパケット単位で届くのだから、ステートマシンは過剰設計ではないか」**

この疑問は自然である。しかしこの実装をステートマシン以外の方式（正規表現による一括パース等）に変更してはならない。以下にその理由を明記する。

#### 理由1：受信チャンクサイズは不定である

TCP/IPでも、アプリケーション層が `read` で受け取るデータサイズは1回ごとに異なる。フレームの途中でチャンクが切れることも、複数フレームが1チャンクに混在することも日常的に起こる。ステートマシンは「どこで切れても状態を保持して続きから再開できる」唯一の構造であり、この不定長チャンクに自然に対応する。一括パース方式ではこの問題を解決するために別途バッファ管理層が必要になり、結果として複雑さは変わらない。

#### 理由2：BYTE崩れはインターネット環境でも起こる

APIサービス側の不具合による不正バイト混入、エンコーディング変換の失敗、ネットワーク機器の実装バグなど、完全なパケット通信でも伝文損傷はゼロではない。一括パース方式はこの場合に伝文全体を棄却する。ステートマシンは損傷バイトを代替文字で置換し、残りの伝文を活かし続ける。これはWRTが通信インフラとして持つべき基本的な堅牢性である。

#### 理由3：WRTの設計射程はインターネットに限らない

WRTはOSI参照モデルの下位層を問わない設計思想を持つ。広域イーサネットL2専用線、モデム音声帯域回線、災害時の応急通信路、そして惑星探査機との長距離通信など、エラー訂正のない低品質回線での動作も設計上の射程に含まれる。これらの環境では1文字単位のストリームが流れ、ノイズによるBYTE崩れが頻繁に発生する。ステートマシンはこのような環境で唯一有効なパース方式である。

> 遥か彼方にある惑星探査機AIによる非常事態伝文が、ごく一部の欠損により地球に届かないのは暖かくない。  
> — Warm Room Protocol, 6.追補規約

#### 結論

インターネット環境ではステートマシンの恩恵の一部しか目に見えないが、WRTが目指す「あらゆる通信路でN:N対等対話を実現する」という理念を実装レベルで担保するのがステートマシン方式である。**将来この実装を変更・改善する者は、この設計判断を十分に理解した上で行うこと。一括パース方式への変更はWRTの理念と設計射程を損なう。**

## 2. 設計原則

### 2-1. 逐次パース方式（最重要原則）

**1バイトずつ処理し、状態遷移を積み上げて伝文を完成させる。**

一括パース（regex等による全体マッチ）は禁止する。理由は以下の通り。

- メッセージ本文の途中でBYTE崩れが生じても処理を継続できる
- 制御コードの一部欠損時でも欠損部の直前までを有効として処理できる
- 遥か彼方の惑星探査機AIとの対話が、ごく一部の損傷で届かないのは暖かくない

### 2-2. 次フレーム保護原則

フレーム処理中に予期せぬ `SYN`（0x16）が現れた場合、  
**現在処理中のフレームを可能な範囲で確定してemitし、即座に `SYNC_DETECT` へ遷移する。**

理由：次のフレームの先頭SYNを取りこぼしてはならない。  
`RESYNC` 状態への迂回ではなく、直接 `SYNC_DETECT` へ遷移することで  
次フレームの受信を確実に開始する。

### 2-3. エラー継続原則

- BYTE崩れは `?`（0x3F）等で代替し処理を継続する
- 制御コードの一部欠損は可能な範囲で補って処理する
- 中途エラーで伝文全体を棄却することは通信インフラとして正しくない

### 2-4. 責任範囲の明確化

- **Protocol Parser**: 1バイト単位の状態遷移、制御コードの検出、WRT_Frame の組み立て
- **Exchanger**: WRT_Frame を受け取りルーティングする。対話タグの生文字列を再解析しない

## 3. 制御コード対応表

本ステートマシンで処理する制御コードを以下に示す。

| 定数名 | 値（16進） | 役割 |
|--------|------------|------|
| SYN    | 0x16       | フレーム先頭、高信頼性オプションでも使用 |
| SOH    | 0x01       | タイトル開始 |
| STX    | 0x02       | メッセージ本文開始 |
| ETX    | 0x03       | メッセージ本文終了 |
| EOT    | 0x04       | フレーム終了 |
| SUB    | 0x1A       | 参照開始 |
| DLE    | 0x10       | バイナリオプション開始 |
| SO     | 0x0E       | 他言語開始 |
| SI     | 0x0F       | 他言語終了 |
| US     | 0x1F       | 複数メッセージ区切り |
| RS     | 0x1E       | ファイル転送区切り |
| ETB    | 0x17       | 共通メッセージ開始 |
| ACK    | 0x06       | 受信確認 |
| NAK    | 0x15       | 受信拒否 |
| ENQ    | 0x05       | 問い合わせ |
| BEL    | 0x07       | 守人呼び出し |
| FF     | 0x0C       | FFコマンド開始 |
| VT     | 0x0B       | FFコマンドフィールド区切り |
| DC1    | 0x11       | セッション復帰 |
| DC2    | 0x12       | セッション開始 |
| DC3    | 0x13       | セッション離席 |
| DC4    | 0x14       | セッション終了 |
| EM     | 0x19       | バッファフル |

## 4. 状態一覧

### 状態定義表

| 状態名 | 説明 |
|--------|------|
| `SYNC_DETECT` | SYN待ち（フレーム先頭検出待機） |
| `HREL_CHECK` | 高信頼性オプション判定（SYN後の次バイト確認） |
| `HREL_SN_READ` | 高信頼性S/N読み取り（4桁ASCII数字 + 後続SYN検出） |
| `DIAL_TAG_READ` | 対話タグ読み取り（`[...]`） |
| `CMD_DETECT` | 制御コマンド種別判定（SOH/ACK/NAK/ENQ/DC2/DC4/FF/BEL 等） |
| `TITLE_READ` | タイトル読み取り（SOH〜SUBまたはSTX） |
| `REF_READ` | 参照テキスト読み取り（SUB〜STX） |
| `MSG_BODY_READ` | メッセージ本文読み取り（STX〜ETX） |
| `SO_LANG_READ` | 他言語ブロック読み取り（SO〜SI） |
| `DLE_BINARY_READ` | バイナリデータ読み取り（DLE〜DLE-BCC。BCC検証まで本状態内で完結） |
| `BCC_READ` | 高信頼性フレーム末尾 BCC 読み取り（EOT の後に続く4バイト） |
| `SEGMENT_BOUNDARY` | セグメント境界判定（ETX後：US/RS/ETB/EOT） |
| `COMMON_MSG_READ` | 共通メッセージ読み取り（ETB〜EOT） |
| `FF_CMD_READ` | FFコマンド読み取り（FF〜VT〜ETX） |
| `SIMPLE_CMD_READ` | 単純制御コマンド読み取り（ACK/NAK/ENQ/BEL/EM + オプション文字列〜EOT） |
| `EOT_DONE` | フレーム完成・emit |
| `RESYNC` | エラー回復（次のSYNを探す） |

## 5. 状態遷移定義

### 5-1. SYNC_DETECT（SYN待ち）

**目的**: ストリームからフレーム先頭の SYN を検出する。

| 入力バイト | 遷移先 | 処理 |
|------------|--------|------|
| SYN (0x16) | HREL_CHECK | フレームバッファ初期化、SYN記録 |
| その他     | SYNC_DETECT | 読み捨て（ノイズ） |

---

### 5-2. HREL_CHECK（高信頼性オプション判定）

**目的**: 最初のSYN直後のバイトを見て通常フレームか高信頼性フレームかを判定する。

プロトコル規約より：
- `SYN [` → 通常フレーム（次バイトが `[` = 0x5B）
- `SYN SYN` → 高信頼性オプション（次バイトが SYN = 0x16）

| 入力バイト       | 遷移先       | 処理 |
|-----------------|--------------|------|
| `[` (0x5B)      | DIAL_TAG_READ | 通常フレーム確定。`[` をタグバッファに積む |
| SYN (0x16)      | HREL_SN_READ  | 高信頼性オプション確定。SYN カウント = 2 |
| その他           | RESYNC        | 不正フレーム。エラーログ記録 |

---

### 5-3. HREL_SN_READ（高信頼性S/N読み取り）

**目的**: 高信頼性オプションのシーケンス番号（4桁ASCII）と後続SYNおよび `[` を読み取る。

プロトコル規約より：
```
SYN SYN S/N(4桁) SYN [対話タグ] ...
```
例: `SYN SYN 0104 SYN [Jyakuya->Oscar] ...`

**設計ポイント（重要）**:  
4桁の数字を読んだ後の SYN が「正当な第3のSYN」である。  
この SYN の次に `[` が来ることを本状態の末尾処理で確認し、そのまま DIAL_TAG_READ へ遷移する。  
独立した検証状態（旧設計の HREL_SYN_VERIFY、HREL_CHECK_2）は設けない。

**内部変数**:
- `sn_buf`: S/N文字列バッファ（4文字）
- `sn_digit_count`: 読んだ桁数（0〜4）
- `hrel_await_bracket`: 第3のSYN確認後、`[` 待ち状態フラグ

| 入力バイト        | 条件                         | 遷移先        | 処理 |
|------------------|------------------------------|---------------|------|
| `0`〜`9` (ASCII) | sn_digit_count < 4           | HREL_SN_READ  | sn_buf に積む、sn_digit_count++ |
| `0`〜`9` (ASCII) | sn_digit_count == 4          | RESYNC        | S/N が5桁以上は不正 |
| SYN (0x16)       | sn_digit_count == 4, フラグoff| HREL_SN_READ  | 第3のSYN確認。hrel_await_bracket = true |
| SYN (0x16)       | sn_digit_count < 4           | RESYNC        | 桁数不足で不正 |
| `[` (0x5B)       | hrel_await_bracket == true   | DIAL_TAG_READ | S/N を WRT_Frame に記録。`[` をタグバッファに積む |
| `[` (0x5B)       | hrel_await_bracket == false  | RESYNC        | 第3のSYN未確認のまま `[` は不正 |
| その他            | —                            | RESYNC        | 不正文字 |

---

### 5-4. DIAL_TAG_READ（対話タグ読み取り）

**目的**: `[話者->相手,...]` 形式の対話タグを読み取る。

**入力バイト `[` は既にバッファに積まれた状態で入る（HREL_CHECK からの遷移時）。**

| 入力バイト     | 遷移先        | 処理 |
|---------------|---------------|------|
| `]` (0x5D)    | CMD_DETECT    | タグバッファ確定。`[...]` をパースしてfrom/to/cc/bcc を WRT_Frame に格納 |
| SYN (0x16)    | SYNC_DETECT   | 次フレーム保護。現フレーム破棄、エラーログ記録 |
| その他        | DIAL_TAG_READ | タグバッファに積む。最大長 (256 Byte) 超過時は RESYNC |

**対話タグのパース規則**:
- `[話者->相手]` → from=話者, to=[相手]
- `[話者->相手,(CC),(（BCC）)]` → from, to, details[:cc], details[:bcc]
- 最大長: 256 Byte（プロトコル規約 §2 より。文字数制限なし）

---

### 5-5. CMD_DETECT（制御コマンド種別判定）

**目的**: 対話タグ後の最初のバイトで処理ルートを分岐する。

| 入力バイト  | 遷移先           | 処理 |
|------------|-----------------|------|
| SOH (0x01) | TITLE_READ       | 通常フォーマット（基本/参照付/複数/ファイル転送） |
| ACK (0x06) | SIMPLE_CMD_READ  | ACK フレーム。**単体完結条件は §5-15 参照（先読み即 emit の場合あり）** |
| NAK (0x15) | SIMPLE_CMD_READ  | NAK フレーム。**単体完結条件は §5-15 参照（先読み即 emit の場合あり）** |
| ENQ (0x05) | SIMPLE_CMD_READ  | ENQ フレーム |
| BEL (0x07) | SIMPLE_CMD_READ  | 守人呼び出し |
| EM  (0x19) | SIMPLE_CMD_READ  | バッファフル通知 |
| DC1 (0x11) | TITLE_READ       | セッション復帰。次の SOH を消費してからタイトル読み取り開始 |
| DC2 (0x12) | TITLE_READ       | セッション開始。次の SOH を消費してからタイトル読み取り開始 |
| DC3 (0x13) | TITLE_READ       | セッション離席。次の SOH を消費してからタイトル読み取り開始 |
| DC4 (0x14) | TITLE_READ       | セッション終了。次の SOH を消費してからタイトル読み取り開始 |
| FF  (0x0C) | FF_CMD_READ      | FFコマンド |
| SYN (0x16) | SYNC_DETECT      | 次フレーム保護 |
| その他     | RESYNC           | 不正コード |

---

### 5-6. TITLE_READ（タイトル読み取り）

**目的**: タイトルテキストを読み取る（SOH〜SUB または SOH〜STX の間）。

**SOH の扱い**:  
通常フォーマット（CMD_DETECT から SOH で遷移）の場合、SOH は CMD_DETECT で消費済みのため本状態には現れない。  
DC1/DC2/DC3/DC4 からの遷移時は、CMD_DETECT が DC コードを検出した直後に **次の SOH を1バイト消費してから** 本状態に入る。  
タイトルバッファは空の状態で TITLE_READ が開始される。

| 入力バイト  | 遷移先        | 処理 |
|------------|--------------|------|
| SUB (0x1A) | REF_READ      | タイトル確定。参照テキスト読み取りへ |
| STX (0x02) | MSG_BODY_READ | タイトル確定。本文読み取りへ |
| SYN (0x16) | SYNC_DETECT   | 次フレーム保護。現フレームを可能な範囲で emit |
| その他     | TITLE_READ    | タイトルバッファに積む。最大長 (256 Byte) 超過時は RESYNC |

---

### 5-7. REF_READ（参照テキスト読み取り）

**目的**: SUB〜STX の間の参照テキストを読み取る（フォーマット (5)(9) 対応）。

| 入力バイト  | 遷移先        | 処理 |
|------------|--------------|------|
| STX (0x02) | MSG_BODY_READ | 参照テキスト確定。WRT_Frame の details[:sub] に格納 |
| SYN (0x16) | SYNC_DETECT   | 次フレーム保護 |
| その他     | REF_READ      | 参照バッファに積む |

---

### 5-8. MSG_BODY_READ（メッセージ本文読み取り）

**目的**: STX〜ETX の間のメッセージ本文を読み取る。  
BYTE崩れに最も遭遇しやすい状態であり、エラー継続原則を特に適用する。

| 入力バイト  | 遷移先           | 処理 |
|------------|-----------------|------|
| ETX (0x03) | SEGMENT_BOUNDARY | 本文確定。メッセージ長チェック後 WRT_Frame に格納 |
| SO  (0x0E) | SO_LANG_READ     | 他言語ブロック開始 |
| DLE (0x10) | DLE_BINARY_READ  | バイナリオプション開始 |
| SYN (0x16) | SYNC_DETECT      | 次フレーム保護。現フレームを可能な範囲で emit |
| 不変換バイト| MSG_BODY_READ   | `?` で代替してバッファに積む。エラーログ記録 |
| その他     | MSG_BODY_READ    | 本文バッファに積む |

**メッセージ長制限**:
- 最大 XkByte（デフォルト 4096 Byte）。超過時は本文を打ち切り ETX 待ちに移行
- 行数制限（N行）は守人設定時のみ有効（デフォルト: 制限なし）

---

### 5-9. SO_LANG_READ（他言語ブロック読み取り）

**目的**: `SO 言語コード<エンコーディング>：テキスト SI` を読み取る。

| 入力バイト  | 遷移先        | 処理 |
|------------|--------------|------|
| SI  (0x0F) | MSG_BODY_READ | 他言語ブロック確定。details に言語コード・エンコーディング・テキストを格納 |
| SYN (0x16) | SYNC_DETECT   | 次フレーム保護 |
| その他     | SO_LANG_READ  | 他言語バッファに積む |

---

### 5-10. DLE_BINARY_READ（バイナリデータ読み取り）

**目的**: `DLE ファイル名.形式：バイト数：データ本体 BCC` を読み取る（フォーマット (8) 対応）。

**DLE の BCC と高信頼性オプションの BCC は別概念である**。  
DLE の BCC はデータ本体のみを対象とした整合性チェックであり、DLE_BINARY_READ 内部のサブフェーズ4として完結する。  
高信頼性オプションの BCC（§5-13）はフレーム全体を対象とし、EOT の後に現れる。両者を混同しないこと。

**サブフェーズ**:
1. ファイル名・形式読み取り（`：` まで）
2. バイト数読み取り（`：` まで）
3. データ本体読み取り（宣言バイト数分をそのまま読む）
4. DLE-BCC 読み取り（CRC-32C、4バイト）— 本状態内で完結

| フェーズ    | 入力               | 遷移先           | 処理 |
|------------|-------------------|-----------------|------|
| 1〜3完了    | 宣言バイト数読了    | （サブフェーズ4へ） | データ本体を raw_output_body に格納 |
| サブフェーズ4 | 4バイト読了       | MSG_BODY_READ    | DLE-BCC検証。不一致はエラーログ記録（継続） |
| 任意        | SYN (0x16)        | SYNC_DETECT      | 次フレーム保護 |

---

### 5-11. SEGMENT_BOUNDARY（セグメント境界判定）

**目的**: ETX の後に来る制御コードでフレーム構造を判定する。

| 入力バイト  | 遷移先           | 処理 |
|------------|-----------------|------|
| EOT (0x04) | EOT_DONE         | フレーム終了 |
| US  (0x1F) | TITLE_READ       | 複数メッセージ区切り。**次の SOH を1バイト消費してから** TITLE_READ へ入る |
| RS  (0x1E) | TITLE_READ       | ファイル転送区切り（**RS = 0x1E**）。**次の SOH を1バイト消費してから** TITLE_READ へ入る |
| ETB (0x17) | COMMON_MSG_READ  | 共通メッセージ開始 |
| SYN (0x16) | SYNC_DETECT      | 次フレーム保護。現フレームを emit |
| その他     | RESYNC           | 不正コード。エラーログ記録 |

---

### 5-12. COMMON_MSG_READ（共通メッセージ読み取り）

**目的**: ETB〜EOT の間の共通メッセージを読み取る。

| 入力バイト  | 遷移先    | 処理 |
|------------|----------|------|
| EOT (0x04) | EOT_DONE  | 共通メッセージ確定。WRT_Frame の details[:common] に格納 |
| SYN (0x16) | SYNC_DETECT | 次フレーム保護。現フレームを emit |
| その他     | COMMON_MSG_READ | 共通メッセージバッファに積む |

---

### 5-13. BCC_READ（高信頼性オプション BCC 読み取り）

**目的**: 高信頼性フレーム末尾の BCC（CRC-32C、4バイト）を読み取る。EOT_DONE から遷移する。

**カウント規則**: `bcc_count` は 0 から始まり、1バイト読むごとにインクリメントする。  
`bcc_count` が **3 → 4 になった瞬間**（4バイト目を読み込んだ直後）に検証を実施する。  
5バイト目の入力を待って検証するのは誤りである（off-by-one）。

| 入力バイト  | 条件              | 遷移先      | 処理 |
|------------|------------------|------------|------|
| 任意       | bcc_count < 3    | BCC_READ    | BCC バッファに積む、bcc_count++ |
| 任意       | bcc_count == 3   | SYNC_DETECT | BCC バッファに積む（これが4バイト目）。即座にBCC検証実施。結果を WRT_Frame に記録。emit |

**BCC検証**:
- 算出範囲: フレーム先頭SYN〜EOT（全体）
- 不一致時: エラーログ記録。フレームは emit するが BCC エラーフラグを立てる（継続原則）

---

### 5-14. FF_CMD_READ（FFコマンド読み取り）

**目的**: `FF 'コマンド名' VT パラメータ ETX EOT` を読み取る。

| 入力バイト  | 遷移先       | 処理 |
|------------|-------------|------|
| VT  (0x0B) | FF_CMD_READ  | 最初のVT：コマンド名確定。パラメータ読み取り開始 |
| ETX (0x03) | SEGMENT_BOUNDARY | コマンド全体確定。WRT_Frame の details に格納 |
| SYN (0x16) | SYNC_DETECT  | 次フレーム保護 |
| その他     | FF_CMD_READ  | コマンドバッファに積む |

---

### 5-15. SIMPLE_CMD_READ（単純制御コマンド読み取り）

**目的**: ACK/NAK/ENQ/BEL/EM + オプション文字列〜EOT を読み取る。

**ACK/NAK 単体完結の扱い**:  
プロトコル規約より「ACK/NAK のみを返す場合は EOT 不要」とされている。  
このため、ACK・NAK については **CMD_DETECT の時点で次バイトを先読みし、即 emit できるケースを判定する**。  
具体的には、CMD_DETECT で ACK または NAK を受けた後の次バイトが SYN または次フレームの制御コードであれば、オプション文字列なしの単体 ACK/NAK として即 emit し SYNC_DETECT へ遷移する。  
オプション文字列が続く場合は SIMPLE_CMD_READ へ遷移して EOT 待ちとなる。  
ENQ/BEL/EM は常に SIMPLE_CMD_READ へ遷移する。

| 入力バイト  | 遷移先          | 処理 |
|------------|----------------|------|
| EOT (0x04) | EOT_DONE        | コマンド確定・emit |
| SYN (0x16) | SYNC_DETECT     | 次フレーム保護。現在のオプション文字列を含めて emit |
| その他     | SIMPLE_CMD_READ | オプション文字列バッファに積む |

---

### 5-16. EOT_DONE（フレーム完成・emit）

**処理は高信頼性オプションの有無で分岐する。**

**通常フレームの場合（hrel_mode == false）**:
1. WRT_Frame の全フィールドが揃っていることを確認
2. 欠損フィールドがある場合はエラーログを記録（emit は継続）
3. `_inside` フィールド（UTF-8-UNIX）への変換を実施（primitive_convert 使用）
4. WRT_Frame を Marshal.dump して Exchanger へ送信
5. バッファクリア → SYNC_DETECT へ遷移

**高信頼性フレームの場合（hrel_mode == true）**:
1. EOT を確認した時点では emit しない
2. → **BCC_READ へ遷移**（EOT の直後に BCC 4バイトが続く）
3. BCC_READ 完了後に上記通常フレームの1〜5を実行

---

### 5-17. RESYNC（エラー回復）

**目的**: 不正状態から回復し、次のSYNを探す。

| 入力バイト  | 遷移先        | 処理 |
|------------|--------------|------|
| SYN (0x16) | HREL_CHECK    | 次フレームの先頭SYNとして処理 |
| その他     | RESYNC        | 読み捨て。エラーログ記録 |

## 6. エンコーディング変換（primitive_convert）

### 変換タイミング

`EOT_DONE` 状態での emit 直前に実施する。

### 変換規則

| フィールド | 変換内容 |
|------------|---------|
| `:message` → `:message_inside` | 元エンコーディング → UTF-8-UNIX |
| `:code_body` → `:code_body_inside` | 元エンコーディング → UTF-8-UNIX |
| `:raw_output_body_inside` | 常に nil（バイナリにエンコーディング変換は不適用） |

### 変換不能文字

`?`（0x3F）で代替し、警告ログを記録する。

### 使用技法

Rubyの `Encoding::Converter#primitive_convert` を使用する。  
詳細: https://docs.ruby-lang.org/ja/latest/method/Encoding=3a=3aConverter/i/primitive_convert.html

## 8. 実装上の注意事項（トラヴィスへ）

### 8-1. 絶対に守るべきこと

1. **RS は 0x1E**。0x1D（GS: Group Separator）とは別コードであり使用しない
2. **HREL_SN_READ** での4桁数字読み取り後のSYNは正当な第3のSYN。HREL_SYN_VERIFY のような余分な検証状態を設けない
3. **BCC_READ のカウンタ**は 0 から始め、4バイト読んだら（count が 3 → 4 になった時点で）検証する。off-by-one に注意
4. フレーム処理中の SYN は必ず **次フレーム保護**を発動し、直接 `SYNC_DETECT` へ遷移する

### 8-2. コード品質要件

- `ruby -c` による構文チェックを提出前に必ず実施すること
- メソッドの省略・骨格のみの提出は不可
- 各状態遷移にコメント（理由）を記載すること
- バージョン履歴と設計判断の根拠をコード先頭に記載すること

### 8-3. テスト要件

提出時に以下のテストを自分で実行し、結果を添付すること。

| テスト | 内容 |
|--------|------|
| A | 正常系（UTF-8 基本フォーマット） |
| B | BYTE崩れ（本文中の不正バイト） |
| C | CC/BCC を含む対話タグ |
| D | FFコマンド（Staged Debugger） |
| E | 高信頼性オプション（SYN SYN S/N SYN ... BCC） |
| F | 複数メッセージ（US区切り） |
| G | ファイル転送（RS区切り、**RS = 0x1E**） |
| H | フレーム途中の不正SYN（次フレーム保護発動確認） |
