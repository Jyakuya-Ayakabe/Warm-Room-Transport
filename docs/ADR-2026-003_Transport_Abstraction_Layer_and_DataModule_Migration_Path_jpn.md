# ADR-2026-003: Transport抽象層の導入とData Module+Signal方式への移行パス保持

**作成**: ルクレツィア  
**承認**: 寂夜（初代守人）  
**決定者**: ルクレツィア（開発責任者）・寂夜（初代守人）による合意  
**決定日**: 2026年4月2日  
**改訂日**: 2026年4月3日（Transport選択肢拡張・Marshal維持の構造的理由追記）  
**ステータス**: 承認済み  
**関連文書**: design_coding_standard_2_6_3, ADR-2026-002（Marshal通信ポリシー）

---

## 決定内容

1. **現行のMarshal+UDSを維持する**（ADR-2026-002を変更しない）
2. **各コンポーネント間の通信にTransport抽象層を一枚挟む**設計とする
3. **Transport実装は4種を用意し、用途・状況に応じて選択可能とする**（後述）
4. 差し替えは特定の条件を満たした場合にのみ実施する（後述）
5. 今回詳細設計・本実装を行うのは、実際に使用する方式のみとする

## 背景と経緯

### 発端

寂夜より「OS-9のデータモジュールを現代Rubyで実現する案」（`/dev/shm` + mmap + Signal）が提示された。「同じマシン内のプロセス間通信になぜ重い処理を挟むのか」という根本的な問いが起点となった。

### 技術的検討

16室同時稼働（人間4名+AI4名、1/4がSD使用）という運用前提のもとで、各方式の性能を比較検討した。

**通常負荷時の転送コスト比較（毎秒約240転送を想定）：**

| 方式 | 1転送コスト | 240転送/秒のCPU消費 |
|------|-----------|-------------------|
| Marshal + UDS | 30〜100μs | 7〜24ms/秒（1〜2%） |
| Data Module + Signal | 5〜18μs | 1.2〜4.3ms/秒（0.2%以下） |

**結論として、通常負荷では両方式に実用上の差はない。**

ボトルネックの優先順位は以下の通り：

1. **PostgreSQL HDD I/O** ── 最大の制約（SSD化で改善可能）
2. **エンコーディング変換（Protocol Parser）** ── CP932←→UTF-8変換が集中する場面
3. **RubyのGC「Stop the World」** ── ハードウェア強化では解決不能（後述）
4. **プロセス間通信** ── 通常負荷では問題にならない

### Marshal維持の構造的理由

WRT_Frameは13フィールドが固定構造だが、**`:details` フィールドだけが動的Hashである**。

固定フィールドは送受信双方が構造を知っているため、`Array#pack / String#unpack` による直接バイナリ化（メタデータ不要・ゼロオーバーヘッド）が原理上可能である。しかし `:details` は中身が毎フレーム異なり、WRTの多様なエンコーディング文化を扱う都合上、JSONのUTF-8強制は理念違反となる。エンコーディング情報を保持したままHashを汎用シリアライズできるのは現時点でMarshalのみである。

`:details` でMarshalを呼ぶ以上、他の13フィールドをpack/unpackで軽量化しても**Marshalの呼び出し自体は1フレームに1回必ず発生する**。中途半端な混在より全フィールドをMarshalで統一した方が実装が簡潔であり、効果対コストが合わない。

**Marshal維持を廃止できる前提条件：`:details` の内容が固定バイナリ構造に置き換えられること。**  
この判断は、開発が進んで `:details` に実際何を入れるかが固まり、かつ実測でMarshalがボトルネックになった時に行えばよい。

### なぜData Module方式を「転んだ時の対策」として残すか

**過去の教訓（寂夜より）**

過去の大規模システム障害事例として、納入された配電網自動化システムが状態変化の集中時に即時追従できず停電事故を繰り返した事例がある。ソフト改修を1年間続けたが解決できずシステム返品・多額賠償となった。この重さ・遅さの大きな要因が**オブジェクトのプロセス間通信と付随するGCだった**。当時の前世代システムはC構造体のメモリ共有+Signalで稼働しており、この問題を持たなかった。

**WRTにおける同様のリスク**

RubyのMarshal.dumpは毎回ヒープ上に一時オブジェクト群を生成する。通常負荷では問題ない。しかし複数室が同時にSD実行結果を受け取る場面等、**状態変化が集中するスパイク時**にGC「Stop the World」が刺さるリスクは原理上排除できない。

RAMを32GBから64GBに増設した場合、ヒープが大きくなり1回あたりのGC停止時間がむしろ**長くなる**可能性もある。すなわちこの問題はハードウェア強化で解決できない性質のものである。

**ハードウェア強化との役割分担**

```
SSD換装     → DB I/O問題を解決
Data Module → GCスパイク問題を解決
```

両者は別の問題への別の対策であり、一方で他方を代替できない。

## OS-9 Data Module方式の特性と利点

WRTが採用しているMarshal+UDS方式との対比として、Data Module方式の特性を記録する。

### 速さ・軽さ

- **ゼロコピー**: 送受信プロセスが同一物理メモリを直接読み書きするため、カーネルバッファへのコピーが発生しない（UDSは2回のコピーが避けられない）
- **GC非介入**: オブジェクト生成を伴わないため、高頻度・集中負荷時もRubyのGC「Stop the World」が発生しない
- **syscall最小**: シグナル通知1回のみ。UDSのwrite/readの往復syscallが不要
- **遅延特性の安定**: スパイク時でも遅延特性が変化しない（Marshalは負荷集中時に遅延が増大する）

### 観測性（OS-9の `mdir` に相当する透明性）

`/dev/shm` 上のファイルはOSの通常ファイルとして存在するため、専用ツールなしに状態観測が可能である。

```bash
# 全チャンネル一覧（mdir相当）
ls -la /dev/shm/wrt_*

# ヘッダ状態のリアルタイム監視（Status Flag・PIDの変化が見える）
watch -n 0.1 hexdump -C -n 16 /dev/shm/wrt_exchanger_sm

# Rubyインスペクタで中身を解析
# → Status=0(空)/1(未読)・WriterPID・ReaderPID・DataSizeが即読み取れる
```

これにより「どのチャンネルでデータが詰まっているか」を、稼働中のシステムに手を入れることなく外部から観測できる。Marshal+UDSでは `socat` でストリームを覗く手間が必要なのに対し、Data Module方式はフラグ1バイトで状態が一目瞭然である。

### 制約

- 同一ホスト内プロセス間専用（ルーター越えには使えない）
- `/dev/shm` のパーミッション管理が必要（モニタリングのしやすさの裏返し）
- Fiddle経由のmmap呼び出しが必要（Ruby標準機能のみでは実装できない）

## 設計方針

### Transport抽象層

各コンポーネント（ExchangerCore, Protocol Parser, Session Manager等）間の通信を、Transportインターフェースを介してのみ行う設計とする。

```ruby
# transport層のインターフェース
module WRT
  class Transport
    def send_frame(frame)  = raise NotImplementedError
    def recv_frame         = raise NotImplementedError
  end

  # 同一ホスト内コンポーネント間（現行方式）
  class UDSTransport < Transport

  # 同一ホスト内コンポーネント間（集中負荷対策・差し替え候補）
  class SHMTransport < Transport

  # WRTサーバー ← → Staged Debugger専用機間（ルーター越えない）
  class UDPTransport < Transport

  # WRTサーバー ← → Emacsクライアント（人間ユーザー、SSHトンネル越え）
  class TCPTransport < Transport
end
```

各コンポーネントはTransportの具体実装に依存しない。設定の変更のみで切り替え可能とする。

**WRT_Frame構造体は変更しない。** Transport層の差し替えはWRT_Frameより下位の層での変更であり、各コンポーネントの実装・インターフェースには影響しない。

### 各Transportの用途と実装方針

| Transport実装 | 用途 | シリアライザ | 実装方針 |
|--------------|------|------------|---------|
| UDSTransport | 同一ホスト内コンポーネント間 | Marshal | **詳細設計・本実装対象** |
| SHMTransport | 同上（集中負荷対策） | Marshal | 発動条件確定後に実装 |
| UDPTransport | WRTサーバー↔SD専用機 | Marshal | **詳細設計・本実装対象** |
| TCPTransport | WRTサーバー↔Emacsクライアント | Marshal | **詳細設計・本実装対象** |

今回の詳細設計・本実装を行うのは**実際に使用するUDS・UDP・TCPの3方式**のみ。SHMTransportは発動条件を満たした時点でルクレツィアが実装する。

## 差し替えの発動条件

以下の条件を満たした場合にData Module方式（SHMTransport）への移行を検討する：

- HDDをSSDに換装した後も、L480での10室同時稼働が安定しない

原因がGCスパイクであるか否かを問わない。プロセス間通信に起因すると判断できる不安定性であれば、Data Module方式を試す根拠として十分とする。SSD換装で解決した場合、SHMTransportは引き続き保持するが発動しない。

## 採用しなかった選択肢

### 「今すぐData Module方式に移行する」

性能上の緊急性がなく、ADR-2026-002の改定コストと実装リスクを現時点で負う合理性がない。

### 「移行パスを用意せず現行のみ維持する」

過去の大規模システム障害事例が示す通り、集中負荷時のGCスパイクは平均値では検知できず、発生してから対処しようとすると手遅れになるリスクがある。今のうちに差し替え可能な設計としておくコストは小さく、保険として合理的。

### 「MessagePackにシリアライザを変更する」

MessagePackはMarshalより5〜7倍高速だが、WRTの負荷水準では0.1%程度のCPU差に過ぎない。また `:message` 等のフィールドをbin型で扱うルールの策定とADR-2026-002改定が必要となる。現時点では優先度が低い。将来シリアライズコストがボトルネックと判明した場合は再検討する。

### 「pack/unpackでMarshalを部分排除する」

WRT_Frameの13フィールドをpack/unpackで軽量化しても、`:details`（動的Hash）でMarshalを呼ぶ以上、1フレームあたりのMarshal呼び出し回数は変わらない。効果対コストが合わず採用しない。Marshal完全排除の前提条件は `:details` の固定バイナリ化である。

## 影響範囲

| 対象 | 影響 |
|------|------|
| WRT_Frame構造体 | **変更なし** |
| Protocol Parser（トラヴィス） | **変更なし** |
| Message Exchanger（オスカー） | **変更なし** |
| PMM・Staged Debugger（ティナーシャ） | **変更なし** |
| design_coding_standard | 次版改定時にTransport抽象層の記述を追加 |
| 各担当への依頼書 | 現行依頼書の変更不要。次回依頼書からTransport層経由を明記 |

Transport抽象層の設計・実装はルクレツィアが単独で行う。他メンバーへの通知は実装完了後。

## 備考

本ADRは「転んだ時の対策を今のうちに用意する」という方針に基づく。性能問題が顕在化する前に保険を設計しておくことは、システムが本番稼働に近づくにつれてその価値が増す。

寂夜の言葉：「転ばぬ先の杖ではなく転んだ時の対策の一つ」
